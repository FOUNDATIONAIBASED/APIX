ApiX Gateway — API Reference

REST API v2 · WebSocket agent protocol · Server-Sent Events
Base URL: http://<server-ip>:3000  · All endpoints return JSON

🔑 Authentication

All write endpoints require an API key. Read endpoints work without a key by default (configurable).

ℹ️
Pass the API key in the X-API-Key header, or as Authorization: Bearer <key>, or as ?api_key=<key> query parameter.
Example Request
curl -X POST http://localhost:3000/api/v1/messages/send \
  -H "X-API-Key: apix_abc123..." \
  -H "Content-Type: application/json" \
  -d '{"to":"+15551234567","body":"Hello from ApiX!"}'
Create your first API key (via console or terminal)
# Via curl (from same server, no auth required to create first key)
curl -X POST http://localhost:3000/api/v1/apikeys \
  -H "Content-Type: application/json" \
  -d '{"name":"My App","permissions":["*"]}'

# Response: { "key": "apix_...", "id": "key_..." }
# SAVE THE KEY — it will never be shown again!
Permissions
PermissionDescription
*Full access to all endpoints
messages:readRead messages, conversations
messages:writeSend SMS/MMS, bulk send
devices:readRead device list and status
devices:writeApprove, suspend, delete devices
webhooks:writeCreate and manage webhooks

✉️ Send SMS / MMS
POST /api/v1/messages/send Send a single SMS or MMS
Request Body
FieldTypeRequiredDescription
tostringrequiredRecipient phone number (E.164 or local)
bodystringrequired*Message text (* or use template_id)
fromstringoptionalSender number. Auto-selects from pool if omitted
typesms|mmsoptionalDefault: sms
media_urlstringoptionalURL of media attachment (MMS)
template_idstringoptionalTemplate ID — body is rendered from this
variablesobjectoptionalTemplate variables e.g. {"first_name":"John"}
device_idstringoptionalForce a specific device to send
Request Example
POST /api/v1/messages/send
X-API-Key: apix_...
Content-Type: application/json

{
  "to": "+15551234567",
  "body": "Hello {first_name}!",
  "template_id": "tpl_abc123",
  "variables": { "first_name": "Alice" }
}
Response (202 Accepted)
{
  "success": true,
  "msgId": "msg_abc123xyz",
  "to": "+15551234567",
  "from": "+15559876543",
  "deviceId": "dev_abc123",
  "status": "queued"
}
Opt-out Error (422)
{ "error": "+15551234567 has opted out.", "code": "OPT_OUT" }
📦 Bulk Send
ℹ️
Bulk send queues messages asynchronously. Opt-out numbers are automatically filtered. Max 10,000 messages per request.
POST /api/v1/messages/bulk Send up to 10,000 messages in one API call
Request Body
{
  "messages": [
    { "to": "+15551234567", "body": "Hi Alice!" },
    { "to": "+15559876543", "body": "Hi Bob!", "variables": { "name": "Bob" } }
  ],
  "from": "+15550001111",      // optional sender override
  "template_id": "tpl_abc123" // optional — renders body from template for each row
}
Response (202 Accepted)
{
  "success": true,
  "queued": 1998,
  "skipped": 2,
  "skipped_details": [
    { "to": "+15559991111", "reason": "opted_out" }
  ]
}

🔗 Webhooks

Configure URLs to receive real-time event notifications. Supports JSON, Twilio-compatible, and n8n formats with optional HMAC-SHA256 signatures.

🔒
Webhook requests include the header X-ApiX-Signature: sha256=... (and X-Twilio-Signature alias) when a secret is configured. Verify it with: HMAC-SHA256(secret, requestBody)
Available Events
message.inbound    — inbound SMS/MMS received
message.outbound   — outbound message queued
message.delivered  — delivery confirmed
message.failed     — message failed to send
message.status     — any status change
device.online      — device connected
device.offline     — device disconnected
device.registered  — new device registered
campaign.started   — campaign started
campaign.completed — campaign completed
optout.received    — STOP keyword received
*                  — all events
POST /api/v1/webhooks Create webhook endpoint
Request
{
  "name": "n8n Automation",
  "url": "https://n8n.example.com/webhook/apix",
  "events": ["message.inbound", "optout.received"],
  "secret": "my-secret-key",
  "format": "n8n"   // json | twilio | n8n
}
Payload Formats
JSON (default)
{ "event": "message.inbound", "timestamp": "...",
  "data": { "id": "msg_...", "from": "+1555...", "to": "+1555...", "body": "..." } }
Twilio format
{ "MessageSid": "msg_...", "From": "+1555...", "To": "+1555...", "Body": "...",
  "NumSegments": "1", "MessageStatus": "received" }
n8n format
{ "headers": { "x-apix-event": "message.inbound" },
  "body": { "event": "...", "data": { ... } } }
POST /api/v1/webhooks/:id/rules Add a filter rule (only dispatch if rule matches)
Request — available operators: equals, not_equals, contains, not_contains, starts_with, ends_with, regex
{ "field": "data.from", "operator": "starts_with", "value": "+1555" }
// Only fire webhook if "from" starts with +1555

{ "field": "data.body", "operator": "regex", "value": "STOP|UNSUBSCRIBE" }
// Only fire if body matches regex
POST /api/v1/webhooks/:id/test Send a test event to the webhook 
{ "success": true, "message": "Test event dispatched" }
🤖 LLM / AI Auto-reply

Connect Ollama, LM Studio, LocalAI, OpenRouter, Groq, Together AI, or any OpenAI-compatible endpoint. Supports weighted load balancing across multiple instances and per-contact conversation context.

✅
Supported types: ollama · openai · lmstudio · localai · groq · together · openrouter · custom (OpenAI-compat)
POST /api/v1/llm/instances Add an LLM backend instance
Ollama example
{
  "name": "Local Llama3",
  "type": "ollama",
  "base_url": "http://localhost:11434",
  "model": "llama3:8b",
  "system_prompt": "You are a helpful SMS assistant. Keep replies under 160 chars.",
  "weight": 1
}
LM Studio / OpenAI-compat example
{
  "name": "LM Studio",
  "type": "lmstudio",
  "base_url": "http://localhost:1234",
  "model": "lmstudio-community/Meta-Llama-3-8B-Instruct-GGUF",
  "weight": 2
}
OpenRouter / Groq / cloud example
{
  "name": "Groq Mixtral",
  "type": "openai",
  "base_url": "https://api.groq.com/openai",
  "model": "mixtral-8x7b-32768",
  "api_key": "gsk_...",
  "weight": 3,
  "timeout_ms": 10000
}
POST /api/v1/llm/rules Create auto-reply rules for inbound messages 
// Auto-reply ALL inbound messages
{ "name": "Reply all", "trigger_type": "all", "llm_id": "llm_abc123", "auto_reply": true }

// Reply only if body contains "help"
{ "name": "Help desk", "trigger_type": "keyword", "trigger_value": "help",
  "llm_id": "llm_abc123", "auto_reply": true }

// Reply only to specific sender number
{ "name": "VIP", "trigger_type": "from", "trigger_value": "+15551234567",
  "llm_id": "llm_abc123", "auto_reply": true }

// Use regex
{ "name": "Order queries", "trigger_type": "regex", "trigger_value": "order|track|ship",
  "llm_id": "llm_abc123", "auto_reply": true }
POST /api/v1/llm/chat Direct chat API (picks healthiest instance automatically) 
POST /api/v1/llm/chat
{ "message": "What's the weather like?", "session_id": "+15551234567" }
// session_id is used to maintain conversation context per user

{ "reply": "I'm an SMS assistant. How can I help you today?",
  "instance_id": "llm_abc123", "model": "llama3:8b" }

🚫 Opt-outs / STOP List
⚠️
Opt-outs are automatically processed when a device receives STOP, STOPALL, or UNSUBSCRIBE. Numbers on the opt-out list are automatically blocked from outbound messages.
POST /api/v1/optouts/check Check if a number is on the opt-out list 
POST /api/v1/optouts/check
{ "number": "+15551234567" }
→ { "number": "+15551234567", "opted_out": true }
DELETE /api/v1/optouts/:number Re-subscribe — remove from opt-out list 
DELETE /api/v1/optouts/+15551234567
→ { "success": true }

📡 Server-Sent Events

Real-time event stream. Connect once and receive all events as they happen.

GET /api/v1/events SSE stream — connect and receive live events
JavaScript Example
const es = new EventSource('http://localhost:3000/api/v1/events');
es.addEventListener('message:inbound', e => {
  const data = JSON.parse(e.data);
  console.log('SMS from', data.from, ':', data.body);
});
es.addEventListener('device:online', e => {
  console.log('Device came online:', JSON.parse(e.data).deviceId);
});
Event types
message:inbound   message:status   message:queued
device:online     device:offline   device:registered
optout:received

🔄 Twilio Compatibility
✅
ApiX Gateway accepts Twilio-formatted webhook payloads and provides a Twilio-compatible message endpoint, making it a drop-in replacement for Twilio in many integrations.
POST /api/v1/messages/twilio-compat Twilio REST API-compatible send endpoint 
// Same field names as Twilio
{ "To": "+15551234567", "From": "+15550001111", "Body": "Hello!" }

// Response matches Twilio MessageInstance structure
{ "sid": "msg_...", "from": "+15550001111", "to": "+15551234567",
  "body": "Hello!", "status": "queued", "direction": "outbound-api",
  "num_segments": "1", "date_created": "2026-03-14T10:00:00.000Z" }
Using the Twilio Node SDK
const twilio = require('twilio');
// Point the client at your ApiX Gateway
const client = twilio('AC_apix', 'your-api-key', {
  edge: null,
  httpClient: { /* override baseUrl */ },
});
// Or simply call the REST endpoint directly with form-encoded Twilio params
Webhook format when receiving from Twilio-compatible systems
// When you set a webhook format to "twilio", outgoing payloads use Twilio field names:
{
  "MessageSid": "msg_abc123",
  "From": "+15551234567",
  "To": "+15559876543",
  "Body": "Hello!",
  "NumSegments": "1",
  "NumMedia": "0",
  "MessageStatus": "received"
}
