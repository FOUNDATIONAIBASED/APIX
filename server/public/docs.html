ApiX Gateway — API Reference

REST API v2 · WebSocket agent protocol · Server-Sent Events
Base URL: http://<server-ip>:3000  · All endpoints return JSON

🖥️ Server & setup

This page documents the Node.js gateway shipped in server/. Agents connect over WebSocket; operators use the browser UI at / and this reference at /docs.

✅
First run: cd server && npm install && cp .env.example .env — set JWT_SECRET and HMAC_SECRET — then node src/index.js or npm start. Open the UI, complete the setup wizard if prompted, then create API keys under the console.
Check that the process is up
curl -s "$(echo "${BASE:-http://127.0.0.1:3000}")/api/v1/status" | head -c 400

Useful fields include smtp_configured, smtp_profile_count, and smtp_routing_mode when debugging email.

👤 Sessions vs API keys

Use the right credential for the job — mixing them up is the most common integration mistake.

CredentialTypical useHow it’s sent
Session cookieWeb console; admin-only routes such as /api/auth/email-smtp/*Browser stores apix_session after login; or send the session token if your client supports it
API keyScripts, backends, mobile apps calling /api/v1/...X-API-Key or Authorization: Bearer <key>
Log in and capture the session cookie (curl)
# Replace host and credentials
BASE="http://127.0.0.1:3000"
curl -s -c apix_cookies.txt -X POST "$BASE/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"your-password"}'

# Use the cookie on admin routes
curl -s -b apix_cookies.txt "$BASE/api/auth/me"
🔐
If the user has 2FA enabled, POST /api/auth/login returns requires_2fa: true and a challenge_token (no session cookie yet). Complete login with:
POST /api/auth/2fa/verify · body {"token":"<challenge_token>","code":"123456"} — on success the response sets apix_session like a normal login.

Copy, paste, replace host and keys. In production mode, never put API keys in URLs.

🔑 Authentication

All write endpoints require an API key. Read endpoints work without a key by default (configurable).

ℹ️
Pass the API key in the X-API-Key header, or as Authorization: Bearer <key>. Query-string API keys are not supported. Browser sessions use a CSRF cookie (apix_csrf) + X-CSRF-Token header on mutating requests.

🔗 Webhooks

Configure URLs to receive real-time event notifications. Supports JSON, Twilio-compatible, and n8n formats with optional HMAC-SHA256 signatures.

🔒
Webhook requests include the header X-ApiX-Signature: sha256=... (and X-Twilio-Signature alias) when a secret is configured. Verify it with: HMAC-SHA256(secret, requestBody)
Available Events
message.inbound    — inbound SMS/MMS received
message.outbound   — outbound message queued
message.delivered  — delivery confirmed
message.failed     — message failed to send
message.status     — any status change
device.online      — device connected
device.offline     — device disconnected
device.registered  — new device registered
campaign.started   — campaign started
campaign.completed — campaign completed
optout.received    — STOP keyword received
*                  — all events
POST /api/v1/webhooks Create webhook endpoint
Request
{
  "name": "n8n Automation",
  "url": "https://n8n.example.com/webhook/apix",
  "events": ["message.inbound", "optout.received"],
  "secret": "my-secret-key",
  "format": "n8n"   // json | twilio | n8n
}
Payload Formats
JSON (default)
{ "event": "message.inbound", "timestamp": "...",
  "data": { "id": "msg_...", "from": "+1555...", "to": "+1555...", "body": "..." } }
Twilio format
{ "MessageSid": "msg_...", "From": "+1555...", "To": "+1555...", "Body": "...",
  "NumSegments": "1", "MessageStatus": "received" }
n8n format
{ "headers": { "x-apix-event": "message.inbound" },
  "body": { "event": "...", "data": { ... } } }
POST /api/v1/webhooks/:id/rules Add a filter rule (only dispatch if rule matches)
Request — available operators: equals, not_equals, contains, not_contains, starts_with, ends_with, regex
{ "field": "data.from", "operator": "starts_with", "value": "+1555" }
// Only fire webhook if "from" starts with +1555

{ "field": "data.body", "operator": "regex", "value": "STOP|UNSUBSCRIBE" }
// Only fire if body matches regex
POST /api/v1/webhooks/:id/test Send a test event to the webhook 
{ "success": true, "message": "Test event dispatched" }

📧 Email / Multi-SMTP

Session cookie auth (apix_session) · admin only. Configure multiple outbound SMTP identities with per-window quotas (UTC), routing, and quota warning emails to admins.

⚠️
Max profiles default 25, cap 100 — set SMTP_MAX_PROFILES in server .env. If any profile is saved in the database, legacy SMTP_* env vars are not used for sending.
Concepts (read this first)
  • Profile — One SMTP endpoint (host, credentials, from-address, optional quotas).
  • Routing — How the server picks which profile sends the next mail: fallback (by priority, skip exhausted or failing profiles), round_robin, or least_used (fewest sends in the current UTC hour).
  • Limits — Per profile, per window: hourly, daily, weekly, monthly. Use 0 for unlimited in that window. All buckets are UTC.
  • Warnings — When usage crosses limit_warn_threshold_pct (default 80%) or a cap is hit, admins can receive email (deduped). Extra addresses: admin_notify_emails; plus admins with an email on their user record.
Profile object — field reference
FieldTypeDescription
idstringStable id (e.g. primary); used in profile_id for targeted tests
enabledbooleanfalse skips the profile entirely
namestringHuman label in UI/logs
host / portstring / numberSMTP server; port defaults to 587 if omitted
securebooleanTypically false on 587 + STARTTLS; true for implicit TLS (e.g. 465) if your provider requires it
user / passstringAuth credentials (passwords are masked in GET responses)
fromstringEnvelope / default From address
from_namestringDisplay name (default ApiX Gateway)
reply_tostringOptional Reply-To
poolbooleanConnection pooling hint for nodemailer transport
tls_reject_unauthorizedbooleanDefault true; set false only for dev/labs with self-signed SMTP
prioritynumberLower runs earlier in fallback mode (ties broken by array order)
limitsobject{ hourly, daily, weekly, monthly } — each 0 = unlimited
End-to-end: login → save config → test (curl)
BASE="http://127.0.0.1:3000"
COOKIE_JAR="apix_cookies.txt"

# 1) Admin login (stores apix_session)
curl -s -c "$COOKIE_JAR" -X POST "$BASE/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"YOUR_PASSWORD"}'

# 2) Replace profiles with your real SMTP (example: two lanes)
curl -s -b "$COOKIE_JAR" -X PUT "$BASE/api/auth/email-smtp/config" \
  -H "Content-Type: application/json" \
  -d '{
    "routing_mode": "fallback",
    "limit_warn_threshold_pct": 80,
    "admin_notify_emails": "ops@example.com",
    "profiles": [
      {
        "id": "main",
        "enabled": true,
        "name": "Primary SMTP",
        "host": "smtp.example.com",
        "port": 587,
        "secure": false,
        "user": "smtp-user",
        "pass": "smtp-secret",
        "from": "noreply@example.com",
        "from_name": "ApiX Gateway",
        "priority": 10,
        "limits": { "hourly": 50, "daily": 500, "weekly": 0, "monthly": 0 }
      }
    ]
  }'

# 3) Inspect masked config + max_profiles
curl -s -b "$COOKIE_JAR" "$BASE/api/auth/email-smtp/config" | head -c 800

# 4) Usage vs limits
curl -s -b "$COOKIE_JAR" "$BASE/api/auth/email-smtp/usage"

# 5) Send test mail through all enabled profiles (or one)
curl -s -b "$COOKIE_JAR" -X POST "$BASE/api/auth/email-smtp/test" \
  -H "Content-Type: application/json" -d '{}'
curl -s -b "$COOKIE_JAR" -X POST "$BASE/api/auth/email-smtp/test" \
  -H "Content-Type: application/json" -d '{"profile_id":"main"}'

🔄 Twilio Compatibility
✅
ApiX Gateway accepts Twilio-formatted webhook payloads and provides a Twilio-compatible message endpoint, making it a drop-in replacement for Twilio in many integrations.
POST /api/v1/messages/twilio-compat Twilio REST API-compatible send endpoint 
// Same field names as Twilio
{ "To": "+15551234567", "From": "+15550001111", "Body": "Hello!" }

// Response matches Twilio MessageInstance structure
{ "sid": "msg_...", "from": "+15550001111", "to": "+15551234567",
  "body": "Hello!", "status": "queued", "direction": "outbound-api",
  "num_segments": "1", "date_created": "2026-03-14T10:00:00.000Z" }
